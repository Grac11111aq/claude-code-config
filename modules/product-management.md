## 🎯 市場分析・競合調査

### 市場調査フレームワーク

```yaml
TAM/SAM/SOM分析:
  TAM（Total Addressable Market）: 理論上の最大市場
  SAM（Serviceable Addressable Market）: 実際に狙える市場
  SOM（Serviceable Obtainable Market）: 現実的に獲得可能な市場

3C分析:
  Customer（顧客）: ニーズ・行動・セグメント
  Competitor（競合）: 強み・弱み・戦略
  Company（自社）: 資源・能力・ポジション

PEST分析:
  Political: 政治・法規制
  Economic: 経済状況・市場動向
  Social: 社会・文化・ライフスタイル
  Technological: 技術・イノベーション
```

### 競合分析実践

```yaml
競合調査項目:
  基本情報:
    - 設立年・規模・資金調達状況
    - ビジネスモデル・収益構造
    - 主要機能・サービス範囲
  
  戦略分析:
    - ターゲット顧客・ポジショニング
    - 価格設定・プロモーション
    - 技術スタック・アーキテクチャ
  
  強み・弱み:
    - ユーザーレビュー分析
    - 機能比較・差別化要因
    - 市場シェア・成長率

LLM活用例:
  調査プロンプト:
    "以下の競合について包括的な分析を行ってください：
     - [競合サービス名]
     
     分析観点：
     1. ビジネスモデル・収益構造
     2. 主要機能・特徴
     3. ターゲット顧客層
     4. 価格設定戦略
     5. 技術的優位性
     6. 市場での評価・ユーザーフィードバック
     7. 成長戦略・将来展望
     
     最新の情報を含めて詳細に報告してください。"
```

## 🧭 プロダクト戦略策定

### プロダクトビジョン・ミッション

```yaml
ビジョン設定:
  - 5-10年後の目指す姿
  - 社会・業界への影響
  - ユーザーの理想的な体験

ミッション定義:
  - 解決する課題の明確化
  - 提供価値の具体化
  - 差別化要因の明文化

例:
  ビジョン: "すべての開発者が最高品質のソフトウェアを効率的に作れる世界"
  ミッション: "AI支援により開発効率を10倍向上させ、品質問題を90%削減する"
```

### プロダクトロードマップ

```yaml
時間軸別計画:
  3ヶ月（短期）:
    - MVP開発・リリース
    - 初期ユーザー獲得
    - 基本的なPMF検証
  
  1年（中期）:
    - 機能拡張・改善
    - 市場シェア拡大
    - 収益化モデル確立
  
  3年（長期）:
    - 市場リーダーポジション
    - 新市場・新機能展開
    - エコシステム構築

フィーチャー優先順位:
  評価軸:
    - ユーザー価値（1-5点）
    - ビジネス価値（1-5点）
    - 開発コスト（1-5点、逆）
    - 技術リスク（1-5点、逆）
  
  優先度スコア: (ユーザー価値 + ビジネス価値) / (開発コスト + 技術リスク)
```

## 📊 プロダクトメトリクス・KPI

### 成長メトリクス

```yaml
AARRR（海賊指標）:
  Acquisition（獲得）:
    - 新規ユーザー数
    - 流入チャネル別コンバージョン率
    - 顧客獲得コスト（CAC）
  
  Activation（活性化）:
    - アクティベーション率
    - 初回体験完了率
    - Time to Value（価値実感までの時間）
  
  Retention（継続）:
    - 継続率（1日、7日、30日）
    - 月間アクティブユーザー（MAU）
    - チャーン率（解約率）
  
  Referral（紹介）:
    - 紹介率・バイラル係数
    - Net Promoter Score（NPS）
    - 口コミ・レビュースコア
  
  Revenue（収益）:
    - 月間経常収益（MRR）
    - 顧客生涯価値（LTV）
    - LTV/CAC比率
```

### プロダクト品質メトリクス

```yaml
ユーザビリティ:
  - タスク完了率
  - タスク完了時間
  - エラー発生率
  - ユーザー満足度（CSAT）

技術品質:
  - ページロード時間
  - アプリクラッシュ率
  - API応答時間
  - 稼働率（Uptime）

ビジネス品質:
  - コンバージョン率
  - 平均注文金額（AOV）
  - セッション継続時間
  - ページビュー数
```

### メトリクス実装

```typescript
// プロダクトメトリクス収集システム
export class ProductMetrics {
  private analytics: AnalyticsService;
  
  // ユーザー行動追跡
  trackUserAction(userId: string, action: string, properties?: any): void {
    this.analytics.track({
      userId,
      event: action,
      properties: {
        timestamp: new Date(),
        ...properties
      }
    });
  }
  
  // コンバージョン計測
  trackConversion(userId: string, conversionType: string, value?: number): void {
    this.analytics.track({
      userId,
      event: 'conversion',
      properties: {
        type: conversionType,
        value: value || 0,
        timestamp: new Date()
      }
    });
  }
  
  // 継続率計算
  async calculateRetentionRate(cohortDate: Date, period: number): Promise<number> {
    const cohortUsers = await this.getCohortUsers(cohortDate);
    const retainedUsers = await this.getRetainedUsers(cohortUsers, period);
    
    return retainedUsers.length / cohortUsers.length;
  }
  
  // LTV計算
  async calculateLTV(userId: string): Promise<number> {
    const user = await this.getUser(userId);
    const monthlyRevenue = await this.getMonthlyRevenue(userId);
    const churnRate = await this.getChurnRate(user.segment);
    
    return monthlyRevenue / churnRate;
  }
}
```

## 🎨 プロダクト企画・要件定義

### ユーザーストーリーマッピング

```yaml
ユーザージャーニー:
  1. 認知・発見: どのように知るか
  2. 検討・評価: 何を比較するか
  3. 試用・体験: どう使い始めるか
  4. 利用・習慣化: 継続的にどう使うか
  5. 推奨・拡散: 他者にどう勧めるか

ストーリーマップ作成:
  横軸: ユーザージャーニーの時系列
  縦軸: 機能の重要度・優先度
  
  Epic → Feature → User Story の階層構造
```

### MoSCoW法による優先順位付け

```yaml
Must Have（必須）:
  - プロダクトの核となる機能
  - なければ成立しない要素
  - 法的・規制上の必須要件

Should Have（重要）:
  - ユーザー体験を大幅に改善
  - 競合優位性を提供
  - ビジネス価値が高い

Could Have（あれば良い）:
  - ユーザー満足度を向上
  - 差別化要因になる可能性
  - 実装コストが比較的低い

Won't Have（今回は対象外）:
  - 将来バージョンで検討
  - 現在のリソースでは実現困難
  - 優先度が低い
```

## 💰 収益化・マネタイゼーション

### ビジネスモデル設計

```yaml
サブスクリプション:
  メリット: 予測可能な継続収益
  適用場面: SaaS、継続的価値提供
  価格設定: 価値ベース、競合ベンチマーク

フリーミアム:
  メリット: 低い導入障壁、拡散効果
  適用場面: ネットワーク効果がある
  注意点: 無料ユーザーのコスト管理

マーケットプレイス:
  メリット: スケーラブル、高い収益性
  適用場面: 需要と供給の仲介
  成功要因: 流動性の確保

APIエコノミー:
  メリット: 開発者エコシステム
  適用場面: プラットフォーム戦略
  価格設定: 使用量ベース
```

### 価格戦略

```yaml
価格設定手法:
  コストプラス: コスト + マージン
  価値ベース: 顧客価値に基づく
  競合ベース: 競合価格を参考
  心理的価格: 顧客心理を考慮

プライシング実験:
  A/Bテスト: 異なる価格での比較
  Van Westendorp分析: 価格感度調査
  コンジョイント分析: 機能と価格の最適化
  
段階的価格設定:
  ティア1: 基本機能（エントリー価格）
  ティア2: 標準機能（メイン価格）
  ティア3: プレミアム機能（高付加価値）
```

## 🔄 継続的改善・実験

### A/Bテスト設計

```yaml
実験設計:
  仮説設定: 変更による効果予測
  成功指標: 測定可能なメトリクス
  サンプルサイズ: 統計的有意性確保
  実験期間: 適切な実験期間設定

実装例:
  機能テスト: 新機能 vs 既存機能
  UIテスト: デザインA vs デザインB
  価格テスト: 価格X vs 価格Y
  メッセージテスト: コピーA vs コピーB

統計的分析:
  - 統計的有意性（p < 0.05）
  - 効果サイズの計算
  - 信頼区間の算出
  - 実用的有意性の判断
```
