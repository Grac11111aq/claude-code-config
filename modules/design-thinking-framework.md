# 🧠 デザイン思考プロセス・意思決定フレームワーク

<module_info>
このモジュールは、個人開発者向けのデザイン思考プロセスを定義します。
Design ThinkingとDouble Diamondを統合し、データ駆動での意思決定を重視します。
デザイン・開発兼任者の効率的ワークフローを提供します。
</module_info>

## 🎯 メタアプローチ：手法選択フレームワーク

### Design Thinking vs Double Diamond の使い分け

```yaml
選択基準マトリクス:
  
  Design Thinking適用:
    条件:
      - 新規プロダクト・サービス開発
      - ユーザーニーズが不明確
      - 創造性・イノベーション重視
      - 複雑な社会課題・人間の行動に関わる
    
    特徴:
      - 共感→定義→発想→試作→検証
      - ユーザー中心設計重視
      - 発散的思考・創造性重視
      - 人間の感情・体験重視
  
  Double Diamond適用:
    条件:
      - 既存プロダクト改善・最適化
      - 問題が比較的明確
      - 効率性・体系性重視
      - 技術的制約・実装可能性重視
    
    特徴:
      - 発散→収束→発散→収束
      - 問題探索→解決策実装
      - 論理的・体系的アプローチ
      - 実装可能性重視

統合判断フロー:
  1. プロジェクト特性分析
  2. ユーザー理解度評価
  3. 創造性 vs 効率性の重視度
  4. 手法選択・ハイブリッド適用判断
```

### データ駆動意思決定の優先順位

```yaml
意思決定階層:
  1. データ駆動（最優先）
     - 定量的指標による客観的判断
     - A/Bテスト・ユーザーテスト結果
     - アクセシビリティ検証データ
     - パフォーマンス測定結果
  
  2. 継続的フィードバック
     - 短いサイクルでの検証・改善
     - プロトタイプ・MVP検証
     - ユーザー行動観察
     - 専門家レビュー・LLM活用
  
  3. 自律的判断重視
     - 基本原則に基づく独立判断
     - 経験・直感による補完
     - リスク・コスト・時間制約考慮
     - 完璧主義の回避

データ不足時の対応:
  - 最小限データでの仮説検証
  - 業界標準・ベストプラクティス参照
  - 段階的リリース・フィードバック収集
  - リスク許容範囲での自律判断
```

## 🔄 統合デザインプロセス

### フェーズ1：プロジェクト分析・手法選択

```yaml
1.1 プロジェクト特性分析:
  チェックリスト:
    - [ ] 新規開発 vs 既存改善
    - [ ] ユーザー理解度（既知 vs 未知）
    - [ ] 技術的制約・実装難易度
    - [ ] 創造性 vs 効率性の重視度
    - [ ] 期限・リソース制約
  
  評価指標:
    新規度: 0-10（10=完全新規）
    複雑度: 0-10（10=最高複雑）
    ユーザー理解: 0-10（10=完全理解）
    創造性重視: 0-10（10=最重視）

1.2 手法選択・カスタマイズ:
  Decision Matrix:
    新規度 >= 7 OR ユーザー理解 <= 3
      → Design Thinking ベース
    
    新規度 <= 4 AND 複雑度 <= 6
      → Double Diamond ベース
    
    その他
      → ハイブリッドアプローチ

1.3 プロセス設計:
  - 実行フェーズ・マイルストーン定義
  - データ収集・検証ポイント設定
  - 成功指標・完了基準設定
  - リスク・制約・前提条件整理
```

### フェーズ2A：Design Thinking プロセス

```yaml
2A.1 共感（Empathize）:
  データ収集:
    - ユーザーインタビュー（3-5名）
    - 行動観察・エスノグラフィ
    - 既存データ分析（アナリティクス等）
    - 競合分析・市場調査
  
  LLM活用:
    - ペルソナ生成・検証
    - インタビュー質問設計
    - 行動パターン分析
    - 洞察・仮説生成
  
  成果物:
    - ユーザーペルソナ（詳細版）
    - エンパシーマップ
    - 行動・ニーズ整理
    - 課題・ペインポイント特定

2A.2 定義（Define）:
  問題明確化:
    - ユーザーニーズ・課題統合
    - 問題の根本原因分析
    - 解決すべき課題の優先順位付け
    - Point of View（POV）ステートメント作成
  
  データ検証:
    - 仮説の定量的検証
    - 市場規模・影響度評価
    - 技術的実現可能性評価
    - ビジネス価値・ROI概算
  
  成果物:
    - Problem Statement
    - How Might We（HMW）質問
    - 成功指標・KPI設定
    - 制約条件・前提整理

2A.3 発想（Ideate）:
  アイデア創出:
    - ブレインストーミング（個人・LLM協働）
    - マインドマップ・アフィニティダイアグラム
    - SCAMPER法・強制連想法
    - 既存ソリューション分析・改良
  
  評価・選択:
    - 実現可能性評価（技術・リソース・時間）
    - ユーザー価値評価
    - アクセシビリティ・包摂性評価
    - コスト・リスク評価
  
  成果物:
    - アイデア候補リスト（50-100個）
    - 評価マトリクス
    - 上位3-5個のコンセプト選定
    - 実装戦略・ロードマップ

2A.4 試作（Prototype）:
  プロトタイプ作成:
    - 低忠実度：紙・ワイヤーフレーム
    - 中忠実度：Figma・デジタルモックアップ
    - 高忠実度：インタラクティブプロトタイプ
    - 実装プロトタイプ：MVP・PoC
  
  技術検証:
    - アクセシビリティ検証
    - パフォーマンス検証
    - 実装可能性検証
    - セキュリティ・品質検証
  
  成果物:
    - 各忠実度レベルのプロトタイプ
    - 技術検証結果
    - 実装課題・リスク特定
    - ユーザーテスト準備

2A.5 検証（Test）:
  ユーザーテスト:
    - モデレート・アンモデレートテスト
    - タスクベース・シナリオテスト
    - A/Bテスト・多変量テスト
    - アクセシビリティユーザーテスト
  
  データ分析:
    - 定量データ分析（完了率・時間・エラー）
    - 定性データ分析（発話・行動・感情）
    - 統計的有意性検証
    - 改善点・次期開発計画
  
  成果物:
    - テスト結果レポート
    - 改善優先順位リスト
    - 次期イテレーション計画
    - 学習・洞察まとめ
```

### 意思決定基準・閾値設定

```yaml
必須達成基準（Go/No-Go判定）:
  アクセシビリティ:
    - WCAG 2.1 AA準拠率: 100%
    - 色覚バリアフリー対応: 100%
    - キーボード操作完了率: 100%
    - スクリーンリーダー対応: 100%
  
  機能性・使いやすさ:
    - 主要タスク完了率: 90%以上
    - 重大エラー発生率: 5%未満
    - 平均タスク完了時間: 目標時間の120%以内
    - ユーザー満足度（SUS）: 70以上
  
  パフォーマンス:
    - First Contentful Paint: 1.8秒以内
    - Largest Contentful Paint: 2.5秒以内
    - Cumulative Layout Shift: 0.1未満
    - First Input Delay: 100ms以内

改善優先度マトリクス:
  高優先度（即座対応）:
    - アクセシビリティ基準未達
    - 重大なユーザビリティ問題
    - セキュリティ・品質問題
    - パフォーマンス基準未達
  
  中優先度（次期対応）:
    - ユーザビリティ改善機会
    - 新機能・拡張要望
    - 運用効率化・最適化
    - 技術的負債解消
  
  低優先度（将来検討）:
    - Nice-to-have機能
    - 美的改善・微調整
    - 実験的機能・アイデア
    - 長期的技術革新
```

## 📖 実践ガイド・適用方法

### プロジェクト開始時の手順

1. **プロジェクト特性分析**（30分）
   - 特性分析シートの記入
   - 手法選択・カスタマイズ
   - データ収集・検証計画策定

2. **成功指標・KPI設定**（20分）
   - アクセシビリティ基準確認
   - 機能性・使いやすさ目標設定
   - パフォーマンス・品質基準設定

3. **プロセス・ツール準備**（10分）
   - テンプレート・ツール準備
   - LLM協業戦略策定
   - 記録・文書化システム設定

### 日常的な設計作業での活用

- **毎日の作業開始時**：今日のフェーズ・タスク確認、前日の振り返り
- **意思決定時**：DDRテンプレート活用、データ・根拠の記録
- **LLM協業時**：重要洞察の記録、フォローアップアクション設定
- **作業終了時**：進捗・学習・改善点の記録

### 定期的な見直し・改善

- **週次**：効率性・時間配分の振り返り、プロセス微調整
- **月次**：品質・成果の評価、ツール・テンプレート改良
- **四半期**：手法・戦略の見直し、新技術・トレンド適応
